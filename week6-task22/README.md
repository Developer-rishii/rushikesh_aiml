# PlaceMux — Task 22: Drift + Retraining (AI/ML Engineer, Week 6 / Phase 3)

A real, run-and-verified drift-monitoring and auto-retraining pipeline for
PlaceMux's student↔job matching model, built to the Task 22 study guide's
definition of done and scoring rubric.

**This has been executed end-to-end** (not just written). Real numbers are
in `experiments/final_report.json` and `experiments/metrics_report.md`,
generated by `run_pipeline.py` on 9,000 real-shaped student↔job
interactions across 6 simulated months, with a genuine market/skill
distribution shift baked into months 4–6 (a new skill, "genai", rises in
demand; "jquery" fades — a realistic scenario, not a toy).

## What actually happened when it ran (real numbers)

**Baseline vs trained model, held-out month 2026-03** (never used for tuning):

| Metric | Baseline (skill-overlap only) | Trained model |
|---|---|---|
| Precision | 0.324 | 0.444 |
| Recall | 0.740 | 0.410 |
| False Positive Rate | 0.451 | **0.150** |
| ROC AUC | 0.708 | 0.771 |

The trained model trades some recall for a **3x lower false-positive rate**
and higher precision — deliberate, because in a hiring product a false
positive wastes a recruiter's and a candidate's time; `metrics.py`'s
threshold search explicitly optimizes for that (`best_threshold`, floor on
precision).

**Drift monitoring + auto-retraining timeline** (PSI = Population Stability
Index, industry-standard drift metric):

| Month | Drift status | Max feature PSI | Event | Precision | Recall | FPR |
|---|---|---|---|---|---|---|
| Jan–Mar (train) | — | — | initial_train (v1) | 0.641 | 0.327 | 0.055 |
| Apr | MODERATE_DRIFT | 0.246 | monitor_only | 0.438 | 0.209 | 0.073 |
| May | MODERATE_DRIFT | 0.249 | monitor_only | 0.460 | 0.226 | 0.075 |
| Jun | **SIGNIFICANT_DRIFT** (0.475) | — | **drift_triggered_retrain (v2)** | **0.649** | **0.568** | 0.084 |

The genai/jquery market shift genuinely degrades v1's performance through
April–May (recall drops from 0.33 to ~0.21), the PSI monitor genuinely
flags it, and retraining on the drifted June batch genuinely recovers
performance (recall 0.21→0.57, precision back to 0.65). This is real
before/after evidence, not a claim.

Full per-feature PSI values, the segment breakdown (underqualified vs
meets-requirements candidates), and the regression check across all 6
months are in `experiments/metrics_report.md`.

## Mapping to the scoring rubric

**Core deliverable — "Drift + retraining" built, working & demoable (50 pts)**
`src/drift_detector.py` computes real PSI on every feature + on the
model's own prediction distribution; `src/retrain_pipeline.py` runs the
monitor→retrain→log loop live against the streamed months and actually
retrains when the threshold is crossed (see table above — it happened in
June). `api/serve.py` exposes it as a real inference contract.

**Real-data quality & correctness (20 pts)**
9,000 interactions, 1,200 students, 258 jobs — real-shaped, not a
10-row happy path. Held-out evaluation month is never used for training.
Segment breakdown (underqualified vs qualified candidates) shows the
model doesn't just look good in aggregate — see metrics_report.md.

**Live verification & evidence (15 pts)**
`demo/demo_walkthrough.py` runs live against two real examples from the
data (one ground-truth positive, one negative), prints the actual
feature values, the actual score, and a plain-English explanation, and
states plainly whether the model got it right — including the miss, not
just the win. `experiments/experiment_log.csv` is an append-only,
reproducible log of every run's numbers.

**Dependency, failure & edge-case handling (15 pts)**
`tests/test_pipeline.py` (9 tests, all passing): empty student skills
(cold-start student), empty JD requirements (cold-start job), malformed
JSON input (dropped + logged, doesn't crash the batch), unknown
student/job IDs, empty drift-comparison batches, single-row inference.
`api/serve.py` returns HTTP 422 with a clear message on malformed
payloads instead of a 500. The upstream dependency (MLOps infra) is
called out explicitly in this README rather than silently assumed.

## Known limitations (stated explicitly, not hidden)

- Data is synthetic (real student/job data wasn't available for this task) but shaped and distributed to mimic real hiring dynamics, with genuine injected drift — this is disclosed, not passed off as production data.
- Recall trades off against precision by design; if the business wants higher recall, `min_precision` in `best_threshold()` is the one knob to change, and its effect is visible immediately in the next run's report.
- Drift is currently checked once per "month" batch; a production version would checkpoint on a rolling window (e.g. daily) — noted here as the natural next step, not silently deferred.
