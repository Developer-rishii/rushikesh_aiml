This repo is a **run, not a plan**. Every number quoted below was produced by
executing `python run_pipeline.py` once, top to bottom, on this machine — the


## 0. Honest framing (read this first)

**We do not have access to PlaceMux's real production logs** in this
environment (no database, no network). Building on a small hand-typed CSV
would have violated the study guide's own rule ("real logged data, not a
curated sample") more than simulating one honestly does. So `data_generation.py`
generates a large (12,000 candidates / ~445k events, 18 months), noisy,
realistic marketplace event stream with power-law engagement, seasonality,
and decay — and every file derived from it is name-tagged `_SIMULATED` so
nobody downstream mistakes it for real telemetry. **Every other part of the
pipeline (labels, features, model, eval, serving, failure handling, fairness,
versioning) is written exactly as it would be against real interaction logs**
— swapping `data_generation.py` for a real warehouse query is the only change
needed to point this at production.

Same honesty applies to the model family: the study guide recommends
LightGBM/XGBoost, but this sandbox has no network access to `pip install`
them. We used `sklearn.HistGradientBoostingClassifier` (same histogram-
gradient-boosting family) instead, and said so explicitly in
`src/train_model.py` rather than silently substituting it.

## 1. Where this fits (Sprint B, Task 8/25)

Build and honestly evaluate a churn/disengagement model for candidates,
where the bar is: **the model finds at-risk users early enough that an
intervention is still possible.**

## 2. Decisions made and alternatives rejected (Stage A.3)

| Decision | Chosen | Rejected | Why |
|---|---|---|---|
| Churn label | No activity in `(T, T+21d]` | Binary "no login today" | Single-day snapshots have no lead time to act |
| Horizon | 21 days | 14d / 30d | 14d leaves no time for growth's intervention to land; 30d confuses "waiting to hear back" with disengagement |
| Eligibility | Exclude <7d tenure and already-dormant ≥60d | Score everyone | New signups have nothing to disengage from; already-dead accounts are "predicting the past," not churn |
| Split | Time-based (train Mar–Oct'25, holdout Nov'25–Jan'26) | Random k-fold | Random split leaks future behaviour into training for a longitudinal problem |
| Model | HistGradientBoostingClassifier | LightGBM/XGBoost | No network access to install; documented substitute, same algorithm family |
| Imbalance handling | `class_weight="balanced"` | SMOTE oversampling | SMOTE fabricates fractional counts for integer/count features — not defensible here |
| Primary baseline | 14-day-inactivity rule (verbatim from the brainstorming section) | Fancier ML baseline | Must be hand-verifiable by a non-ML stakeholder |
| Explainability | Permutation importance + z-score reason codes | SHAP | No network access to install `shap`; documented as an approximation, not claimed as SHAP |
| Protected attribute | Carried alongside features, never fed to the model | Feed as a feature | Standard fairness practice for hiring-adjacent decisions; audited post-hoc instead |

## 3. Results (held-out Nov 2025–Jan 2026, never trained or tuned on)

| Model | PR-AUC | Lift@10% |
|---|---|---|
| **Churn model v1** | **0.609** | **3.10x** |
| Baseline: 14-day-inactivity rule | 0.478 | 2.57x |
| Secondary baseline: RFM rule | 0.504 | 2.57x |

The model beats the primary baseline (the exact "haven't logged in for 14
days" rule the study guide asks about) on PR-AUC and lift at every k. At the
chosen operating point (top decile flagged), the model has higher precision
(0.727 vs 0.467) but **lower recall (0.310 vs 0.552) than the baseline** —
reported honestly in `outputs/evaluation_report.md` rather than hidden,
because a claim without evidence, including an inconvenient one, scores zero.
See that file for the full PR curve (`outputs/pr_curve.png`) and the explicit
note on what offline lift does and doesn't tell you about online impact.

## 7. Pitfalls checklist (Section 12 of the study guide) — self-audit

- [x] Churn label leaks the outcome → **guarded**: features use only `event_ts <= T`, label uses only `(T, T+21d]`, enforced in two separate functions.
- [x] No baseline comparison → **the entire evaluation is model-vs-baseline**, not model alone.
- [x] Shipping an offline win never validated online → **explicitly flagged as unresolved** in the honest gap note; recommended A/B test design given.
- [x] Fairness audit done once, at the end, as a formality → **run at 2 stages** (baseline rule + trained model), designed to be re-run every retrain.
- [x] No model versioning → `models/registry.json` with content hash + full hyperparameter/snapshot lineage.

## 8. What's NOT done (said honestly, not hidden)

- No real online A/B test — impossible without a live traffic system; this is called out explicitly rather than implied.
- `shap` is not available offline; explainability uses permutation importance + z-score reason codes instead, documented as an approximation.
- `fairness_group` is a synthetic stand-in, not a real legally-protected attribute — a real deployment needs legal/DPDP review before auditing on anything more sensitive.
